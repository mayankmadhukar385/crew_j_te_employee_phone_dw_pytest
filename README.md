# DataStage → Databricks Conversion Pipeline

## Architecture

```
                    SHARED FRAMEWORK (central, reused by all jobs)
                 ┌──────────────────────────────────────────────────┐
                 │  DeltaManager (SCD 0/1/2/4 + append/upsert/...) │
                 │  AuditLogger · WatermarkManager · DQValidator    │
                 │  ConfigManager · DatabricksJobLogger              │
                 │  SourceReader (JDBC/file/Delta/test_dataframe)    │
                 └────────────────────┬─────────────────────────────┘
                                      │ imports
                 ┌────────────────────┼─────────────────────────────┐
                 │  PER-JOB (thin wrappers, generated by Step 3)    │
                 │  transformations.py + run_pipeline.py + tests/    │
                 │  + config INSERT SQL for etl_job_config table     │
                 └──────────────────────────────────────────────────┘
```

## Commands — Run in Claude Code

### STEP 0 — Discovery
```
Read prompts/step0_prompt.md for your instructions. Analyze the DataStage XML
in input/ folder. Detect all jobs, classify each as sequence or parallel,
extract orchestration logic (conditions, failure paths, parameters), and
write output/discovery.json with the execution plan.
```

### STEP 1 — Generate Lineage
```
Read prompts/step1_prompt.md for your instructions. Read output/discovery.json
for the execution plan. For each parallel job listed in jobs_to_process, parse
it from the XML and generate its lineage Excel (3 tabs: Stage_Sequence,
Mermaid_Lineage, Source_to_Target_Mapping) and standalone Mermaid diagram
(flowchart LR) in output/. If a sequence job exists
(orchestration.sequence_job_name is not null), also generate the orchestration
diagram as a separate .md file using flowchart TB with diamonds for conditions,
dashed arrows for failure paths, START/END circles, and the full exception
handling chain.
```

### STEP 1B — Validate Lineage
```
Read prompts/step1b_prompt.md for your instructions. You are an independent
reviewer. For each parallel job, parse the XML independently, run all 12
validation checks against the generated Excel and Mermaid, fix every gap,
and write corrected files plus validation reports.
```

### STEP 2 — Generate Code Prompts
```
Read prompts/step2_prompt.md for your instructions. For each parallel job,
read the XML + corrected Excel + corrected Mermaid, and generate its
Databricks code prompt with all 10 sections (pipeline identity, stage types,
transformation chain, expression dictionary, column mappings, routing logic,
config template, module structure, test specs, execution order).
```

### STEP 3 — Generate Job Code + Workflow + Config INSERT SQL
```
Read prompts/step3_prompt.md for your instructions. For each parallel job,
generate a PipelineRunner subclass under jobs/{job_name}/ that implements
transform() only — everything else comes from the framework. Also generate
a deploy_config.sql with the INSERT statement for etl_job_config.
If a sequence job exists, generate output/{seq_name}_databricks_workflow.yml
with the complete Databricks Workflow definition: notebook_tasks per job,
condition_tasks for branching, depends_on for task dependencies, job
parameters, and email_notifications for failure handling.
```

### STEP 3B — Validate Code
```
Read prompts/step3b_prompt.md for your instructions. You are an independent
code reviewer. For each parallel job, validate the generated code against
the XML and corrected Excel. Verify it extends PipelineRunner correctly,
uses framework components (no duplication), and the workflow YAML matches
the sequence orchestration. Run all 13 checks. Fix every gap.
```

### STEP 4 — Generate Test Data
```
Read prompts/step4_prompt.md for your instructions. For each parallel job,
generate synthetic test data as PySpark DataFrame factories. Register data
as Spark temp views (source.type = "test_dataframe" in test config). Cover
all routing branches, null handling, error paths, and boundary values.
```

### STEP 5 — Generate Deployment Guide
```
Read prompts/step5_prompt.md for your instructions. Generate DEPLOYMENT.md
covering: framework table DDL (etl_audit_log, etl_job_log, etl_watermark,
etl_job_config), target table DDL, config INSERT SQL execution per job,
Databricks Workflow YAML deployment (merge into databricks.yml), DABs
deploy commands, monitoring queries against audit/log tables, watermark
rollback procedures, and troubleshooting.
```

## Config Management

Config lives in a Delta table — no YAML files, no dual maintenance.

```sql
-- Change SCD type for a job (takes effect on next run, no deployment)
UPDATE catalog.schema.etl_job_config
SET config_json = '{"pipeline": {...}, "target": {"scd_type": 2, ...}}'
WHERE job_name = 'Jb_LMIS_Full_Ld' AND is_active = 'Y';
```

Step 3 generates the initial INSERT SQL. After that, all config
changes are SQL statements or `ConfigManager.update_config_key()`.

## Framework Delta Tables

| Table | Purpose |
|---|---|
| `etl_audit_log` | One start + end row per pipeline execution |
| `etl_job_log` | All operational logs + record-level rejects (partitioned by log_date) |
| `etl_watermark` | Incremental load state per job + source table |
| `etl_job_config` | Versioned job configs as JSON (single source of truth) |

All tables carry `job_id`, `job_name`, `workflow_name` for cross-table joins.
